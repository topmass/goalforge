GoalForge System Map

GoalForge is a local command center on top of Codex. One persistent project memory thread keeps context. Each task gets an isolated worktree and a child Codex thread. The board stores task state, thread IDs, events, context packets, and handoffs.

Command CenterOpenTUI and API server for clicks, status, and board actions.
Main ThreadPersistent Codex memory seeded once per project directory.
Task WorkerChild Codex thread in a git worktree for one feature or fix.
RecoveryMissing Codex rollout becomes a fresh seeded main thread, then retry.
Docs MemoryAGENTS.md, WORKFLOW.md, specsheet, context manifests.

1. Big Picture Flow

The user drives the board from the TUI. GoalForge stores durable state locally and uses the Codex SDK bridge to start, resume, fork, run, steer, review, and absorb work.

2. Persistent Main Thread

The main thread is not just an ID. GoalForge starts it, runs a tiny seed turn, then stores the thread ID. That seed turn gives Codex real history so task workers can fork from it.

Project memory
sequenceDiagram
  participant TUI as OpenTUI
  participant API as GoalForge API
  participant W as Worker.ensureMainThread
  participant C as Codex SDK Bridge
  participant DB as board.sqlite

  TUI->>API: POST /api/main/ensure on startup
  API->>DB: Read project main_thread_id
  alt No stored main thread
    API->>W: ensureMainThread()
    W->>C: thread_start(cwd, model, sandbox)
    C-->>W: threadId
    W->>C: runTurn("GoalForge main thread seed")
    C-->>W: seed turn completed
    W->>DB: Store main_thread_id and summary
  else Thread already stored
    API-->>TUI: Reuse stored thread ID
  end
  API-->>TUI: Project memory panel updates
Why this matters
  • A raw new thread ID may not have a Codex rollout yet.
  • Forking an empty thread can fail with "no rollout found".
  • The seed turn creates a real conversation history.
  • The project thread stays pinned to the folder through .goalforge/board.sqlite.
  • Manual codex continue elsewhere should not replace GoalForge's saved main thread.

4. What Gets Stored Where

The important split is simple: Codex owns opaque thread history. GoalForge owns project identity, board state, task state, docs, context packets, and UI labels.

State map
.goalforge/board.sqlite

Stores task status, run state, events, queued messages, main thread ID, child thread IDs, active turn IDs, touched paths, and blocked reasons.

Codex local sessions

Stores transcript and rollout history. GoalForge treats thread IDs as opaque and resumes or forks through app-server/SDK calls.

Project files

AGENTS.md, WORKFLOW.md, and project-specsheet.md carry durable project behavior and handoffs.

Task artifacts

.goalforge/tasks/TASK-ID/ stores the task packet and manifest that tell workers exactly what context to read.

5. Missing Session Recovery

If Codex says a saved rollout or thread is missing, GoalForge should not ask the user for project input. It rebuilds the project memory thread, clears stale child lineage, and retries.
